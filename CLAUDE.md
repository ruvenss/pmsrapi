# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**PMSRAPI** — a lightweight, framework-less PHP 8.3 REST API + webhook microservice scaffold. One instance = one microservice = one database. It runs on Ubuntu via the PHP built-in server (`php -S`) behind systemd, and is designed to be dropped into an existing directory, self-configure via a wizard, and auto-update itself from GitHub without touching the user's own code.

There is **no build step, no Composer, no test suite, and no application framework**. Everything is procedural PHP using `define()` constants.

> Everything below describes the original **v1** core (in `v1/`). A second,
> modernized core lives in [v2/](v2/) — real REST routing, a DI container, a
> hand-rolled PSR-4 autoloader, mysqli **prepared** statements, and Redis
> (cache, rate-limit, queue, token store). It runs beside v1 and reuses the same
> secret JSON config. **When working in `v2/`, follow [v2/CLAUDE.md](v2/CLAUDE.md)
> and the `.claude/rules/`; do not carry v1's idioms into it, and do not
> "modernize" v1's auto-overwritten core to match v2.**

## Commands

```bash
# Install (root, on the target Linux server) — interactive dialog wizard.
# Generates v1/config.php, writes the secrets JSON to the parent dir,
# installs cron jobs, and deploys the systemd service.
./install.sh

# Deploy / (re)create the systemd service + per-second cron service.
# Run from inside v1/ ; Linux + CLI only.
cd v1 && php deploy.php

# Manually pull the latest core files from GitHub (CLI only).
php update.php

# Run locally for development (from the repo root; serves the whole tree).
php -S 0.0.0.0:8000 -t .
# Endpoints then live at http://localhost:8000/v1/

# Lint — CI runs PHP_CodeSniffer (PSR) on push; run locally if installed:
phpcs v1/
```

There is no way to "run a single test" — no tests exist. Verify changes by curling the running service.

### Calling an endpoint

Every request goes to `/v1/` (not to the endpoint file directly), as JSON, with a Bearer token, and names the target function in the body:

```bash
curl --location --request GET 'http://localhost:8000/v1/' \
  --header 'Content-Type: application/json' \
  --header 'Authorization: Bearer <ms_server_token>' \
  --data '{ "function": "select_rows", "parameters": { "table": "users" } }'
```

## Architecture

### Single dispatcher + function-name routing

[v1/index.php](v1/index.php) is the **only** entry point and does all routing. There is no per-endpoint URL. The flow for every request:

1. Load public config ([v1/config.php](v1/config.php)) and the secrets JSON (`ms_secrets`).
2. Require `Content-Type: application/json` and a `Authorization: Bearer` token; compare it to `ms_server_token` with `hash_equals`.
3. Read the JSON body; require a `function` field.
4. Check the function is whitelisted in `ms_secrets['allowed_functions'][<HTTP_METHOD>]` — **an endpoint file existing is not enough; it must also be listed in config to be callable.**
5. Sanitize the name and `include` `<METHOD>/<function>.php` (e.g. a GET for `select_rows` → [v1/GET/select_rows.php](v1/GET/select_rows.php)).

The HTTP method chooses the folder and the CRUD intent: `GET`=select/read, `POST`=insert/upload, `PUT`=update, `DELETE`=delete.

### Request → response contract

- Request bodies use `parameters` (query/where/table info) and `payload` (bulk/file data) keys — see any endpoint for the shape.
- Endpoints define a function named exactly like the file, then call it at the bottom (`function select_rows(){...} select_rows();`).
- **All responses go through `http_response($code, $data)`** (defined in index.php). It sets standard headers, wraps data as `{"success": bool, "data": ...}`, closes the DB, and **calls `die()`**. Because it terminates, code after a `http_response(...)` call for an error case does not run — this is the framework's control-flow idiom.
- Reserved request constants once routed: `request_method`, `request_body`, `request_data`, `function_path`.

### Config is split in two, and secrets live OUTSIDE the web root

- [v1/config.php](v1/config.php) — **public**, committed-shape constants (`ms_name`, `ms_version`, headers, response map, `ms_logserver`). Generated by the installer.
- `../<ms_name>.json` (path held in the `config_path` constant) — **secret**, loaded into the `ms_secrets` constant: DB credentials, `ms_server_token`, `allowed_functions` whitelist, `env`, `local_log`, and the `universe` list. Stored in the parent directory so it is never web-served or committed.

### Core vs. user code — DO NOT edit core files

The files listed under `source-code` in [manifest.json](manifest.json) are **core framework files**. They are:
- overwritten on every auto-update (`update.php` diffs local vs. remote `manifest.json` version and re-downloads from the `master` branch), and
- listed in the generated `.gitignore` so they never enter the user's repo.

Core files carry a `DO NOT MODIFY / WILL BE OVERWRITTEN` banner (index.php, deploy.php, everything in `general/`, the stock CRUD endpoints, cron scripts). **Put new work in user-owned locations instead:**
- New endpoints: `v1/{GET,POST,PUT,DELETE}/<name>.php` (then add the name to `allowed_functions` in the secrets JSON).
- Shared helpers: `v1/general/custom_functions.php` (auto-included when present).
- Custom translations: `v1/locales/en_custom.json`.
- Event hooks: the per-method `events.php` (see below).

When asked to "add an endpoint," the deliverable is a new function file **plus** the config whitelist entry — one without the other is non-functional.

### Database layer

[v1/general/db.php](v1/general/db.php) (core) opens a single mysqli connection exposed as the `dbconn` constant and provides the query helpers every endpoint uses: `sqlInsert`, `sqlUpdate`, `sqlDelete`, `sqlSelect`, `sqlSelectRow`, `sqlSelectRows`, `sqlSelectPlot`, `sqlCount`, `sqlSum`, plus schema helpers (`getPrimaryKey`, `sqlTables`, `getTableLastUpdateTime`, `sqlTableExist`, `sqlCreateDatabase`). It is included only when `ms_secrets['db']` is set.

> **Note the tension with `.claude/rules/`:** the repo ships a PHP-best-practices rule set (strict types, prepared statements, DI, no globals). The *core* runtime deliberately violates several of these — it is heavily constant/`define()`-based, and the `sql*` helpers build SQL by string concatenation with `mysqli_real_escape_string`, not prepared statements. Follow the `.claude/rules/` conventions in **new user code**; do not "fix" core files to match them (they'll be overwritten, and it breaks the intentionally minimal design).

### Events

Mutating endpoints `include_once` the method's `events.php` (e.g. [v1/POST/insert.php](v1/POST/insert.php) includes `POST/events.php` and calls `after_insert($new_row)`). These hook files are where side effects (logging, webhooks, cross-service calls) belong.

### Webhooks

File-based, no DB. [v1/general/webhooks.php](v1/general/webhooks.php) stores each hook as `webhooks/data/<METHOD>/<uniqid>.json` and dispatches via cURL (`sendWebhook`). The `webhooks/` tree (`data/`, `logs/`, `queue/`) is created on demand and gitignored.

### The "universe" — inter-service calls

A microservice can call sibling services listed in `ms_secrets['universe']` (each has `name`/`ip`/`port`/`token`/`ssl`). `http_rest($node, $function, $payload, $parameters, $method)` in index.php performs the authenticated cross-service call. `universe_*_node` endpoints add/remove nodes by rewriting the secrets JSON atomically (temp file + `copy`).

### Logging

`log_event(...)` (index.php) ships structured events to a remote log server (`ms_logserver`, Bearer `ms_logserver_token`) **only when the event's `env` matches `ms_environment`**, else falls back to a local file. Low-level SQL logging goes to `ms_secrets['local_log']['path']` gated by `local_log.level` (`info` / `errors` / `none`).

### Scheduling

Deploy installs two systemd units per service: the HTTP server (`php -S`) and `<ms_name>_seconds` running [v1/cron/seconds.sh](v1/cron/seconds.sh) (a loop calling `second.php` every ~3s). `install.sh` also registers standard crontab entries for `v1/cron/{minute,hour,day,week,month,year}.php`. Put scheduled work in those files.

## Versioning

Bump the version in **three** places together, or auto-update/`info` will disagree: `manifest.json` (`version` + the `source-code` list if files changed), the `ms_version`/`@version` docblocks in core files, and the release URL/tag. Releases are git tags like `0.0.27`; the installer downloads the tagged zip.
