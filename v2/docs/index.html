PMSRAPI v2 — Deploy & Use Guide

A step-by-step, copy-paste guide to running, using, and deploying v2 microservices. Every API example below runs as-is against the bundled test environment.

Run the examples live Start the test stack (PHP + MariaDB + Redis, seeded with a clients table) and every curl in this guide works verbatim:
bash
cd test-env
docker compose up --build          # serves http://localhost:8080/v2
# in another terminal, export the base URL + token used by every example:
export BASE=http://localhost:8080/v2
export TOKEN=test-token-please-change
For your own server, just change BASE to https://your-host/v2 and TOKEN to your ms_server_token.

PMSRAPI v2 is a framework-less PHP 8.3 microservice core. One instance = one service = one database. It speaks real REST, talks to sibling services over an authenticated NDJSON stream, and uses Redis for caching, rate-limiting, queues and a live debug dashboard.

Prerequisites

NeedWhy / how
PHP 8.3+ with mysqli, redis (phpredis), curl, jsonThe runtime. No Composer, no build step.
MariaDB / MySQLOne database per service (optional — a service with no db block runs fine).
Redis 6+Cache, rate-limit, queue, debug capture, hive registry. If absent, those features no-op gracefully.
Docker (dev only)To run the bundled test environment. Never used for production.

Quick start (2 minutes)

  1. Start the stack.
    bash
    cd test-env && docker compose up --build
  2. Check it's alive (the /health endpoint is public — no token):
    bash
    curl -s $BASE/health
    json
    { "success": true, "data": { "status": "ok", "database": "up", "redis": "up" } }
  3. Make your first authenticated call:
    bash
    curl -s $BASE/info -H "Authorization: Bearer $TOKEN"
  4. List the seeded clients:
    bash
    curl -s "$BASE/clients" -H "Authorization: Bearer $TOKEN"

Configuration

Config is split in two. The public file is committed; the secret file lives outside the web root and holds credentials.

Public — v2/config.php (edit for your service)

php
<?php
return [
    'ms_name'      => 'clients',
    'ms_version'   => '2.0.0',
    // Absolute path recommended in production. Keep it OUTSIDE the web root.
    'secrets_path' => '/etc/pmsrapi/clients.json',
    'headers'      => [ 'Access-Control-Allow-Origin' => '*' /* ... */ ],
];

Secret — the JSON at secrets_path (never commit, never web-serve)

json
{
  "role": "worker",
  "env": "prod",
  "ms_server_token": "CHANGE-ME-long-random-string",
  "local_log": { "path": "/var/log/clients.log", "level": "errors" },

  "db":    { "host": "127.0.0.1", "port": 3306, "name": "clients",
             "username": "clients", "password": "secret" },
  "redis": { "host": "127.0.0.1", "port": 6379, "prefix": "ms:clients:" },

  "cache":      { "ttl": 30 },
  "rate_limit": { "enabled": true, "max": 120, "window": 60 },
  "debug":      { "enabled": true, "auto_arm_on_error": true },

  "resources": {
    "clients": { "table": "clients", "methods": ["GET","POST","PUT","DELETE"], "per_page": 25 }
  }
}
Key idea A database table is only reachable over REST if it appears in resources. That map is your access-control whitelist.

Authentication & the response envelope

Every request (except GET /health) needs a bearer token and, for bodies, JSON:

bash
-H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json"

Every response is wrapped:

json
// success
{ "success": true, "data": { ... }, "meta": { ... } }
// failure
{ "success": false, "error": { "code": "validation_failed", "message": "…", "details": { … } } }

CRUD · Define a resource

Before the CRUD calls work, the table must exist and be listed under resources. The test env already ships this clients table:

sql
CREATE TABLE clients (
  id         INT UNSIGNED  NOT NULL AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(120)  NOT NULL,
  email      VARCHAR(190)  NOT NULL UNIQUE,           -- unique key → upsert target
  status     VARCHAR(20)   NOT NULL DEFAULT 'active',
  balance    DECIMAL(10,2) NOT NULL DEFAULT 0,          -- numeric col for SUM/AVG/MIN/MAX
  created_at TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- seeded rows used by the examples below:
INSERT INTO clients (name,email,status,balance) VALUES
  ('Ann Lee','ann@example.com','active',120.50),
  ('Bob Kim','bob@example.com','active', 80.00),
  ('Cara Ng','cara@example.com','inactive',45.25);
json — add to the secret config
"resources": {
  "clients": { "table": "clients", "methods": ["GET","POST","PUT","DELETE"], "per_page": 25 }
}

CRUD · Create POST

bash
curl -s -X POST "$BASE/clients" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "name": "Dana Poe", "email": "dana@example.com", "status": "active" }'
201 Created
{ "success": true, "data": { "id": 4, "name": "Dana Poe", "email": "dana@example.com",
  "status": "active", "created_at": "2026-07-05 14:00:00" } }
Only real table columns are accepted — unknown keys are dropped, and values must be scalar (or null).

bash

200 OK
{ "success": true, "data": { "id": 4, "name": "Dana Poe", "email": "dana@example.com", "status": "active" } }

CRUD · List, filter, sort & paginate GET

Plain list (paginated)

bash
curl -s "$BASE/clients" -H "Authorization: Bearer $TOKEN"
200 OK
{ "success": true,
  "data": [ { "id": 1, "name": "Ann Lee", ... }, ... ],
  "meta": { "page": 1, "per_page": 25, "total": 3, "total_pages": 1 } }

Filter by any column (equality)

bash
curl -s "$BASE/clients?status=active" -H "Authorization: Bearer $TOKEN"

Sort — order=column or order=column:desc

bash
curl -s "$BASE/clients?order=created_at:desc" -H "Authorization: Bearer $TOKEN"

Paginate — page & per_page (max 200)

bash
curl -s "$BASE/clients?page=2&per_page=50" -H "Authorization: Bearer $TOKEN"

Everything at once

bash
curl -s "$BASE/clients?status=active&order=name&page=1&per_page=10" \
  -H "Authorization: Bearer $TOKEN"
Query paramMeaning
<column>=valueEquality filter on any real column
order=col / order=col:descSort ascending / descending
page, per_pagePagination (1-based; per_page capped at 200)

CRUD · Update PUT / PATCH

Both replace the columns you send (partial update). Only supplied columns change.

bash — full/partial via PUT
curl -s -X PUT "$BASE/clients/4" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "status": "inactive" }'
bash — same via PATCH
curl -s -X PATCH "$BASE/clients/4" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "name": "Dana P." }'
200 OK — returns the updated row
{ "success": true, "data": { "id": 4, "name": "Dana P.", "status": "inactive", ... } }

CRUD · Delete DELETE

bash
curl -s -i -X DELETE "$BASE/clients/4" -H "Authorization: Bearer $TOKEN"
204 No Content
HTTP/1.1 204 No Content     # empty body on success; 404 if the id doesn't exist

CRUD · Stream / bulk export GET

Stream an entire table as NDJSON (one JSON object per line), paged internally so memory stays flat for millions of rows:

bash
curl -sN "$BASE/stream/clients" -H "Authorization: Bearer $TOKEN"
application/x-ndjson
{"id":1,"name":"Ann Lee","email":"ann@example.com","status":"active"}
{"id":2,"name":"Bob Kim","email":"bob@example.com","status":"active"}
{"id":3,"name":"Cara Ng","email":"cara@example.com","status":"inactive"}

A DB-free demo stream is handy for testing the pipe: curl -sN "$BASE/stream/_demo?count=5" -H "Authorization: Bearer $TOKEN"

CRUD · Advanced SELECT — GROUP BY, aggregates, CONCAT POST

For grouped/aggregated reads, POST a structured query spec to /{resource}/query. It supports GROUP BY, aggregates (count/sum/avg/min/max), GROUP_CONCAT, CONCAT, HAVING, DISTINCT, filtering, ordering and limits — all with identifiers whitelisted against the live schema and values bound.

Why structured, not raw SQL The spec is validated field-by-field so there is no SQL-injection surface — an illegal column or function is rejected with 422. For anything the builder can't express, write a custom endpoint in routes.php.

Every example below was captured from a live server on the seeded data — the responses are exactly what you'll get.

1 · Group & count, with GROUP_CONCAT

Count clients per status, and list the names in each group:

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
    "select": [
      { "column": "status" },
      { "fn": "count", "as": "total" },
      { "fn": "group_concat", "column": "name", "separator": ", ", "as": "names" }
    ],
    "group_by": ["status"],
    "order": "total:desc"
  }'
200 OK · verified
{ "success": true, "data": [
  { "status": "active",   "total": 2, "names": "Ann Lee, Bob Kim" },
  { "status": "inactive", "total": 1, "names": "Cara Ng" }
] }

2 · SUM / AVG / MIN / MAX (with a WHERE filter)

Aggregate the balance of active clients only:

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
    "select": [
      { "fn": "sum", "column": "balance", "as": "total" },
      { "fn": "avg", "column": "balance", "as": "average" },
      { "fn": "min", "column": "balance", "as": "lowest" },
      { "fn": "max", "column": "balance", "as": "highest" },
      { "fn": "count", "as": "n" }
    ],
    "where": { "status": "active" }
  }'
200 OK · verified
{ "success": true, "data": [
  { "total": "200.50", "average": "100.250000", "lowest": "80.00", "highest": "120.50", "n": 2 }
] }
Types Decimal aggregates return as JSON strings (to keep precision); counts return as numbers. That's exactly what the server sent.

3 · CONCAT with bound literals

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
    "select": [ { "concat": ["name", { "literal": " <" }, "email", { "literal": ">" }], "as": "label" } ],
    "where": { "email": "ann@example.com" }
  }'
200 OK · verified
{ "success": true, "data": [ { "label": "Ann Lee <ann@example.com>" } ] }

4 · DISTINCT

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "select": [ { "column": "status" } ], "distinct": true, "order": "status" }'
200 OK · verified
{ "success": true, "data": [ { "status": "active" }, { "status": "inactive" } ] }

5 · HAVING — filter the groups

Keep only groups with more than one member:

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
    "select":   [ { "column": "status" }, { "fn": "count", "as": "total" } ],
    "group_by": ["status"],
    "having":   { "total": { "op": ">", "value": 1 } }
  }'
200 OK · verified
{ "success": true, "data": [ { "status": "active", "total": 2 } ] }

6 · Injection attempts are rejected

A malicious column name never reaches SQL — it fails identifier validation:

bash
curl -s -X POST "$BASE/clients/query" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "select": [ { "column": "id); DROP TABLE clients;--" } ] }'
422 · verified (nothing runs)
{ "success": false, "error": { "code": "validation_failed", "message": "Validation failed",
  "details": { "errors": { "column": "Unknown column: id); DROP TABLE clients;--" } } } }

The query spec

FieldShapeNotes
select (required)array of itemseach item is one of the three below
· column{ "column": "x", "as": "…" }a plain column
· aggregate{ "fn": "count|sum|avg|min|max|group_concat", "column": "x", "distinct": false, "separator": ", ", "as": "…" }count without a column → COUNT(*)
· concat{ "concat": ["col", {"literal":"txt"}, …], "as": "…" }literals are bound params
where{ "col": value }equality (AND-ed)
group_by["col", …]whitelisted columns
having{ "alias": { "op": ">", "value": 5 } }op ∈ = != <> > >= < <=; targets an alias or column
order"alias:desc"by alias or column
limit, offset, distinctint / int / bool

CRUD · Upsert — “if exists then update” POST

Insert a row, or update the existing one on a duplicate key, in a single atomic statement (INSERT … ON DUPLICATE KEY UPDATE). The table needs a PRIMARY or UNIQUE key for the conflict to be detected — here, email is unique.

1 · First call — the email is new, so it inserts

bash
curl -s -X POST "$BASE/clients/upsert" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "values": { "name": "Zoe", "email": "zoe@example.com", "status": "active", "balance": 10 } }'
201 Created · verified
{ "success": true, "data": {
  "action": "inserted", "id": 4,
  "record": { "id": 4, "name": "Zoe", "email": "zoe@example.com",
              "status": "active", "balance": "10.00", "created_at": "2026-07-05 20:20:24" }
} }

2 · Same email again — now it updates in place (same id)

bash
curl -s -X POST "$BASE/clients/upsert" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "values": { "name": "Zoe Q", "email": "zoe@example.com", "status": "vip", "balance": 50 } }'
200 OK · verified
{ "success": true, "data": {
  "action": "updated", "id": 4,
  "record": { "id": 4, "name": "Zoe Q", "email": "zoe@example.com",
              "status": "vip", "balance": "50.00", "created_at": "2026-07-05 20:20:24" }
} }

3 · Only change some columns on conflict — the update list

Here only status is overwritten on a duplicate; name is left as it was (verified):

bash
curl -s -X POST "$BASE/clients/upsert" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{ "values": { "name": "ignored", "email": "zoe@example.com", "status": "active" },
        "update": ["status"] }'
actionHTTPMeaning
inserted201no matching key — a new row was created
updated200key existed — the row was updated in place (same id)
unchanged200key existed and values were already identical

Plugins · What & why

A plugin is a self-contained folder under v2/plugins/ that adds your own controllers, models and routes to a running service — without editing a single core file (no bootstrap.php, no routes.php, nothing under src/). Two developers can each own a plugin in the same service and never step on each other: every plugin gets its own namespace and its own URL prefix.

In-process, by design A plugin runs inside this service's process — a fatal error in a plugin takes the service down. It's a code-organisation tool, not a sandbox. If you need true isolation, split the work into a separate microservice (see Multiple microservices).

The shipped Example plugin proves the wiring — it's live in the test env right now:

bash
curl -s "$BASE/example/hello/World" -H "Authorization: Bearer $TOKEN"
200 OK
{ "success": true, "data": { "greeting": "Hello, World!", "from": "example plugin" } }

Plugin routes ride the same pipeline as core routes, so they're behind bearer auth and rate limiting automatically — no extra work.

Plugins · Create one (wizard)

A bash wizard scaffolds a complete, working plugin. Run it from the repo root and answer a couple of prompts:

bash
bash v2/plugin.sh                # interactive: create a plugin
bash v2/plugin.sh list           # what's installed + enabled
bash v2/plugin.sh check Billing  # lint & validate one plugin
bash v2/plugin.sh remove Billing # delete one (asks to confirm)
  1. Run bash v2/plugin.sh and enter a name (e.g. Billing).
  2. Optionally include a database example (it injects the shared core Repository).
  3. The wizard writes the files, lints them, and prints the exact curl to try.

The generated plugin answers immediately at its own prefix — nothing to register:

bash
curl -s "$BASE/billing/ping" -H "Authorization: Bearer $TOKEN"
200 OK
{ "success": true, "data": { "plugin": "billing", "status": "ok" } }

Plugins · Anatomy & the contract

Discovery is by convention: a directory Billing/ must define the entry class Plugins\Billing\BillingPlugin at src/BillingPlugin.php.

v2/plugins/Billing/
plugin.json                 # metadata + "enabled" flag
src/                        # autoloaded as  Plugins\Billing\…
  BillingPlugin.php         # REQUIRED entry class
  Controllers/
    InvoiceController.php
  Models/  Support/         # anything you like
assets/                     # read-only static files (optional)
README.md

The entry class has two hooks and takes no constructor arguments:

php — src/BillingPlugin.php
<?php
declare(strict_types=1);

namespace Plugins\Billing;

use Plugins\Billing\Controllers\InvoiceController;
use Pmsrapi\V2\Core\Container;
use Pmsrapi\V2\Database\Repository;
use Pmsrapi\V2\Http\Request;
use Pmsrapi\V2\Http\Response;
use Pmsrapi\V2\Plugin\AbstractPlugin;
use Pmsrapi\V2\Plugin\PluginRegistrar;
use Pmsrapi\V2\Plugin\PluginRouter;

final class BillingPlugin extends AbstractPlugin
{
    // 1) Add YOUR services. Inject core ones (Repository, …) via $c —
    //    never open your own DB/Redis handle.
    public function register(PluginRegistrar $r): void
    {
        $r->singleton(InvoiceController::class,
            static fn(Container $c): InvoiceController
                => new InvoiceController($c->get(Repository::class)));
    }

    // 2) Add routes. Auto-prefixed with the slug → served at /v2/billing/…
    public function routes(PluginRouter $router, Container $c): void
    {
        $router->get('/invoices/{id}', static fn(Request $req, array $p): Response
            => $c->get(InvoiceController::class)->show($p['id']));
    }
}
The slug is your URL namespace Every route is prefixed with the plugin's slug (its lowercased directory name), so Billing's get('/invoices') is served at GET /v2/billing/invoices. A slug that clashes with a core prefix (info, health, webhooks, stream, hive, …) or a configured resource is rejected at boot.
json — plugin.json (all fields optional)
{
  "name": "Billing",
  "description": "Invoicing endpoints.",
  "version": "1.0.0",
  "author": "You",
  "enabled": true,
  "dependencies": []
}

Set "enabled": false to switch a plugin off without deleting it.

Plugins · Rules & governance

Plugins follow the same discipline as the core — these are enforced or checked, not just suggested:

RuleHow it holds
Database access only via the coreInject Connection/Repository — you get prepared statements + schema-whitelisting for free. Never open your own handle.
Add services, never replacePluginRegistrar throws at boot if you register an id the core (or another plugin) already owns.
Own URL namespaceRoutes auto-prefixed with the slug; core/resource collisions fail loudly at boot.
Auth & rate-limit inheritedPlugin routes ride the same Kernel pipeline as core routes.
No writable state in the code treeCaches → Redis, logs → local_log.path — same as the core.
PHP 8.3, strict types, typed everythingSame rules as v2/src/; validate with bash v2/plugin.sh check <Name>.
Two hooks, then frozen The core integrates plugins in exactly two lines — bootstrap.php registers their services and routes.php mounts their routes. That footprint never grows: the core stays frozen and auto-updatable, and v2/plugins/ is yours and survives updates. Full reference: v2/plugins/README.md.

Multiple microservices · Concepts

Each service owns a set of named functions. A function name must belong to exactly one service — get_client lives in the clients service, and every other service calls it instead of duplicating the logic.

Config keyPurpose
roleworker (normal service) or hive_mind (dev-only coordinator)
functionsThe functions THIS service owns → how to invoke each
universeThe sibling nodes it can reach: name/ip/port/token/ssl
function_mapname → { service, method, path, stream } for functions it CALLS (baked from the Hive)

Example topology: two workers + a dev hive

json — clients-service secret config
{
  "role": "worker",
  "ms_server_token": "clients-token",
  "db": { "host": "127.0.0.1", "name": "clients", "username": "clients", "password": "…" },
  "redis": { "host": "127.0.0.1", "prefix": "ms:clients:" },
  "resources": { "clients": { "table": "clients" } },
  "functions": {
    "get_client":     { "method": "GET", "path": "/clients/{id}" },
    "export_clients": { "method": "GET", "path": "/stream/clients", "stream": true }
  }
}
json — billing-service secret config (calls clients)
{
  "role": "worker",
  "ms_server_token": "billing-token",
  "db": { "host": "127.0.0.1", "name": "billing", "username": "billing", "password": "…" },
  "redis": { "host": "127.0.0.1", "prefix": "ms:billing:" },
  "resources": { "invoices": { "table": "invoices" } },
  "functions": { "get_invoice": { "method": "GET", "path": "/invoices/{id}" } },

  "universe": [
    { "name": "clients", "ip": "10.0.0.11", "port": 8001, "token": "clients-token", "ssl": false }
  ],
  "function_map": {
    "get_client": { "service": "clients", "method": "GET", "path": "/clients/{id}" }
  }
}

Multiple microservices · Call another service

Inside billing, add a custom route that fetches a client from the clients service. Custom routes go at the bottom of v2/routes.php and survive core updates:

php — v2/routes.php
use Pmsrapi\V2\Cluster\ServiceClient;

// GET /v2/billing-report/42  → pulls client 42 from the clients service
$router->get('/billing-report/{id}', static fn(Request $r, array $p): Response =>
    Response::ok(
        $container->get(ServiceClient::class)->call('get_client', ['id' => $p['id']])
    ));

ServiceClient::call() resolves get_client from the local function_map, finds the clients node in universe, authenticates with that node's token, and returns its data. No Hive is contacted at runtime.

bash — from a client of billing
curl -s "https://billing.example.com/v2/billing-report/42" \
  -H "Authorization: Bearer $BILLING_TOKEN"

Multiple microservices · Stream between services

For bulk data, stream instead of buffering. ServiceClient::stream() yields records lazily (constant memory) as they arrive over NDJSON:

php — inside any endpoint / cron / worker
$client = $container->get(ServiceClient::class);

foreach ($client->stream('export_clients', ['status' => 'active']) as $row) {
    // process one client at a time — the whole table never sits in memory
    reindex($row);
}

Multiple microservices · The Hive & the map

Development only The Hive is a coordinator you run in dev. Workers never need it at runtime. Set "role": "hive_mind" and give it every worker in its universe.

Each worker publishes what it owns:

bash
curl -s "https://clients.example.com/v2/capabilities" -H "Authorization: Bearer $CLIENTS_TOKEN"

The Hive collects them, builds the global map, and — crucially — flags any function claimed by two services:

bash — on the hive
curl -s -X POST "$HIVE/v2/hive/refresh" -H "Authorization: Bearer $HIVE_TOKEN"   # poll every worker
curl -s "$HIVE/v2/hive/map"         -H "Authorization: Bearer $HIVE_TOKEN"
curl -s "$HIVE/v2/hive/collisions"  -H "Authorization: Bearer $HIVE_TOKEN"
json — a collision (two services own get_client → FIX IT)
{ "success": true, "data": { "collisions": [
  { "function": "get_client", "owners": ["clients", "billing"] }
] } }
See it graphically Open https://your-hive/v2/hive in a browser for a VueFlow graph of every service and its functions, with duplicates highlighted in red.

Multiple microservices · Bake the map (go standalone)

Deploy a single microservice (production)

Target: Ubuntu 24.04. We deploy a service called clients on 127.0.0.1:8001 behind nginx + TLS.

  1. Install the runtime.
    bash (root)
    add-apt-repository -y ppa:ondrej/php && apt update
apt install -y php8.3-cli php8.3-mysql php8.3-curl php8.3-redis php8.3-mbstring \
               mariadb-server redis-server nginx
  2. Get the code.
    bash
    git clone https://github.com/ruvenss/pmsrapi.git /srv/clients
chown -R www-data:www-data /srv/clients      # app lives at /srv/clients/v2
  3. Create the database + table.
    bash
    mysql -e "CREATE DATABASE clients CHARACTER SET utf8mb4;
CREATE USER 'clients'@'localhost' IDENTIFIED BY 'secret';
GRANT ALL ON clients.* TO 'clients'@'localhost';"
mysql clients -e "CREATE TABLE clients (
  id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY, name VARCHAR(120) NOT NULL,
  email VARCHAR(190) NOT NULL UNIQUE, status VARCHAR(20) NOT NULL DEFAULT 'active',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP);"
  4. Write the secret config outside the web root, locked down.
    bash
    mkdir -p /etc/pmsrapi
cat > /etc/pmsrapi/clients.json <<'JSON'
{
  "role": "worker", "env": "prod",
  "ms_server_token": "REPLACE-with-64-random-chars",
  "local_log": { "path": "/var/log/clients.log", "level": "errors" },
  "db":    { "host": "127.0.0.1", "port": 3306, "name": "clients",
             "username": "clients", "password": "secret" },
  "redis": { "host": "127.0.0.1", "port": 6379, "prefix": "ms:clients:" },
  "rate_limit": { "enabled": true, "max": 120, "window": 60 },
  "resources": { "clients": { "table": "clients", "methods": ["GET","POST","PUT","DELETE"] } }
}
JSON
chown www-data:www-data /etc/pmsrapi/clients.json && chmod 600 /etc/pmsrapi/clients.json
  5. Point the public config at it. Edit /srv/clients/v2/config.php:
    php
    'ms_name'      => 'clients',
'secrets_path' => '/etc/pmsrapi/clients.json',   // absolute
  6. Create the systemd service (the HTTP server) and the webhook worker.
    bash — /etc/systemd/system/clients.service
    cat > /etc/systemd/system/clients.service <<'UNIT'
[Unit]
Description=clients microservice (pmsrapi v2)
After=network.target mariadb.service redis-server.service

[Service]
Environment=PHP_CLI_SERVER_WORKERS=8
ExecStart=/usr/bin/php -S 127.0.0.1:8001 /srv/clients/v2/server.php
WorkingDirectory=/srv/clients
User=www-data
Group=www-data
Restart=always

[Install]
WantedBy=multi-user.target
UNIT
    bash — /etc/systemd/system/clients-worker.service (webhook/queue)
    cat > /etc/systemd/system/clients-worker.service <<'UNIT'
[Unit]
Description=clients webhook worker
After=redis-server.service

[Service]
ExecStart=/usr/bin/php /srv/clients/v2/worker.php
WorkingDirectory=/srv/clients
User=www-data
Restart=always

[Install]
WantedBy=multi-user.target
UNIT
  7. Start everything.
    bash
    systemctl daemon-reload
systemctl enable --now clients.service clients-worker.service
systemctl status clients.service --no-pager
  8. Expose it with nginx + TLS.
    bash — /etc/nginx/sites-available/clients
    cat > /etc/nginx/sites-available/clients <<'NGINX'
server {
    listen 80;
    server_name clients.example.com;

    location / {
        proxy_pass http://127.0.0.1:8001;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_buffering off;                 # let NDJSON stream through
    }
    # never serve the secret json or logs
    location ~* \.(json|log)$ { deny all; }
}
NGINX
ln -s /etc/nginx/sites-available/clients /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx
certbot --nginx -d clients.example.com          # adds HTTPS
  9. Verify.
    bash
    curl -s https://clients.example.com/v2/health
curl -s https://clients.example.com/v2/clients \
  -H "Authorization: Bearer REPLACE-with-64-random-chars"
Hardening Keep the code tree read-only and all writable state (cache, queue, logs) in Redis or /var/log — never inside /srv/clients. Because connections are persistent, size MariaDB max_connections for workers × instances.

Deploy multiple microservices

Deploy each service exactly like the single case, but on its own port, directory, database and secret file. Then wire them together.

1. One unit per service, distinct ports

ServiceDirPortSecretPublic URL
clients/srv/clients8001/etc/pmsrapi/clients.jsonclients.example.com
billing/srv/billing8002/etc/pmsrapi/billing.jsonbilling.example.com

Repeat the single-service steps for billing with 8002, its own DB, and a billing.service/billing-worker.service pair.

2. Route each with its own nginx server block

nginx
server { server_name clients.example.com; location / { proxy_pass http://127.0.0.1:8001; proxy_buffering off; } }
server { server_name billing.example.com; location / { proxy_pass http://127.0.0.1:8002; proxy_buffering off; } }

Prefer subdomains (above). If you must share one host, route by path prefix and strip it:

nginx — path-based
location /clients/ { rewrite ^/clients/(.*) /$1 break; proxy_pass http://127.0.0.1:8001; }
location /billing/ { rewrite ^/billing/(.*) /$1 break; proxy_pass http://127.0.0.1:8002; }

3. Wire the universe

In each service that calls another, list the sibling in universe and add the function_map entry (see Concepts). Do it by hand, or generate it with a dev Hive and php v2/hive-sync.php (bakes it into the secret file for you).

4. (Optional) One-box dev stack with Compose

To run several services + MariaDB + Redis together locally, mirror test-env/docker-compose.yml: one app-style service per microservice on its own port, sharing the db and redis services. That stack is dev/test only — production uses the systemd + nginx recipe above.
